# 开发计划

## 总体目标
本项目的长期目标不是只做一个简单回测脚本，而是逐步演进为一个：
- 面向普通股民也容易使用的 A 股辅助决策工具
- 同时支持 CLI、图形界面、AI Agent 集成
- 先支持手工持仓分析与换仓建议
- 后续可扩展到量化 API / 券商 API 接入
- 支持多数据源（免费 / 付费 / API Key / 兼容代理）切换
- 支持用户自定义参数、策略模板、回测和执行流程

## 当前进展同步

### 已完成
1. **数据源可靠性增强第一阶段**
   - AkShare 增加重试与缓存降级能力
   - 增加 `doctor-data` 健康检查命令
   - 业务命令失败时输出 `provider_diagnostics` 与 `hint`
   - 支持 `--offline` 仅缓存模式

2. **多数据源兼容能力**
   - 已支持 `provider: akshare`
   - 已支持 `provider: tushare`
   - `TushareProvider` 已支持 `tushare` / `tinyshare` 两种 SDK 后端
   - token / 授权码 / SDK 选择均可配置持久化

3. **CLI / Agent 友好性增强**
   - 结构化输出已稳定为 `schema_version/status/message/data`
   - 错误输出中补充 provider 诊断信息
   - 模板导出与工作区初始化能力已接入

4. **普通用户体验增强**
   - UI 支持模板说明
   - UI 支持数据源健康检查入口
   - 新手模式 / 高级模式已具备基础可用性

## 当前阶段重点

### Phase 7：多数据源扩展与可靠性增强
目标：
- 让免费源在波动环境下更能用
- 让付费/兼容数据源更容易配置和诊断
- 让命令行和 UI 都能解释“为什么失败”
- 摆脱“所有 provider 都必须提供 AkShare 风格板块/实时接口”的隐含假设

关键任务：
1. 增加缓存预热命令（如 `warm-cache`）
2. 增加缓存状态查看（如 `cache-status`）
3. 规划主数据源 + 备用数据源机制
4. 继续细化 provider 错误分类与恢复策略
5. 在 UI 中增加更明确的离线/缓存状态提示
6. 持续完善 capability-aware fallback，让 Tinyshare/Tushare 在能力受限时仍能完成轻量筛选、回测和计划

### Phase 9：高级交互层继续推进
目标：
- 让 CLI 更适合 AI Agent 集成
- 让 UI 更适合普通用户长时间使用

关键任务：
1. 保持 JSON schema 稳定
2. 继续增强失败提示与结果摘要
3. 为未来 API 层预留结构化服务输出

## 面向用户的长期能力目标

### 普通用户模式
- 手工录入持仓
- 自动生成分析与换仓建议
- 本地保存持仓与计划
- 简单安装、简单使用、界面清晰
- 在免费源不稳定时仍能通过缓存/离线模式继续使用

### 进阶用户模式
- 自定义参数
- 自定义策略模板
- 更稳定的数据源
- 多版本回测比较
- 可选 Tushare / Tinyshare 等增强数据源

### 量化接入模式
- 支持外部 API Key
- 支持账户读取
- 支持交易执行适配层
- 支持模拟盘 / 实盘切换

## 技术路线建议

### 数据源
- 短期：AkShare + 重试/缓存/离线/诊断
- 中期：增强 Tushare / Tinyshare 兼容能力
- 中长期：支持主备 provider 与按功能路由
- 长期：评估更多商业数据源与机构级方案

### 账户体系
- 短期：本地 JSON + 手工输入
- 中期：统一账户抽象层
- 后期：接券商 / 量化 API

### UI
- 短期：继续使用 Streamlit
- 中期：增加数据源状态、缓存状态、诊断提示
- 后期：根据需求评估独立前端或桌面端方案

### CLI
- 短期：保持易用 + 结构化输出
- 中期：增强诊断、离线、缓存预热能力
- 后期：支持 AI Agent 更稳定调用
